## Summary

- **What problem does this PR solve?** `scroll_only` only supported vertical `swipeDirection: "up" | "down"` with no distance/path control. `scroll_and_resolve_ui_target` had divergent iOS vs Android implementations.
- **Why is this change needed now?** The flat `swipeDirection` enum ages poorly — a structured `gesture` model with optional viewport-relative ratios gives deterministic control needed for stable automation.
- **Why is this the right scope for one reviewable PR?** Two focused contract changes: (1) gesture model for `scroll_only`, (2) remove iOS `scroll_and_resolve_ui_target` and `scroll_and_tap_element` to unify platforms. Both touch the same contract/adapter/test surfaces.

## Capability impact

- Capability category: action
- User-visible or AI-facing behavior change: `scroll_only` now requires `gesture: { direction, startRatio?, endRatio? }` instead of optional `swipeDirection`. `scroll_and_resolve_ui_target` and `scroll_and_tap_element` are Android-only.
- Platforms/frameworks affected: Android / iOS
- Support boundary change: contract-ready — new `ScrollOnlyGesture` interface with typed direction and optional precision ratios
- Invocation guidance impact: canonical — `docs/guides/ai-agent-invocation.zh-CN.md` updated with Android-only boundaries and iOS fallback paths
- Capability truth source checked: `packages/contracts/src/types.ts`, `packages/adapter-maestro/src/capability-model.ts`
- Public docs / canonical guide update: yes — `docs/guides/ai-agent-invocation.zh-CN.md`, `docs/guides/ui-stabilization-timing.md`, `docs/testing/ios-tool-probe-checklist.md`

## Implementation notes

- Key implementation changes: Added `ScrollOnlyGesture` interface, `ScrollOnlyGestureMode` type, `buildScrollOnlySwipeCoordinates()` for 4-direction + ratio support, `normalizeScrollOnlyGesture()` validation. Added `--scroll-only` CLI with gesture flags.
- Contract / policy / session / evidence changes: `ScrollOnlyInput.gesture` required, `swipeDirection` removed from input and output. `ScrollOnlyData.gestureApplied` is single source of truth.
- Deterministic path and fallback behavior: no fallback — `gesture` is required, validation errors return `CONFIGURATION_ERROR`.
- AI-first capability guideline consulted: yes
- Compatibility, migration, or rollout notes: existing `swipeDirection` callers must migrate to `gesture: { direction }`. iOS `scroll_and_resolve_ui_target` / `scroll_and_tap_element` callers must use `scroll_only → wait_for_ui → resolve_ui_target` patterns.
