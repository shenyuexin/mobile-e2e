## Summary

- **What problem does this PR solve?** Four real risks identified in code review of `77140d74..971092ce` (P24-C navigate_back stabilization + pageIdentity): (1) `pageIdentity` silently changed `JSON.stringify`-based state comparison in `perform_action_with_evidence`, retry, and checkpoint logic; (2) two incompatible tree hash implementations (`derivePageIdentity` vs `waitForUiStable`) shared an algorithm but had different input sources — no documentation of non-cross-comparability; (3) `navigate_back` over-inferred `stateChanged: false` from matching page tree hashes and smuggled `stableFingerprint` into `postBackPageIdentity.treeHash`, violating the PageIdentity contract; (4) `wait_for_ui_stable` was missing from `capability-model.ts`, README tool lists, and server tests. Additionally, a pre-existing bug in `verifyResolvedIosPointWithHooks` silently overrode verification mismatch to `verified: true` when `exitCode === 0`, `preBackTreeHash` was captured AFTER the back action (post-action-early snapshot) instead of BEFORE it, and navigate-back had no tool-level behavioral tests.
- **Why is this change needed now?** These are blocking changes — merging the previous commits without fixes would silently alter orchestration semantics for all users of `pageIdentity` and `navigate_back`, and the CI was failing on the pre-existing verifyResolvedIosPoint bug.
- **Why is this the right scope for one reviewable PR?** All fixes address the same review feedback set and touch closely related files: `session-state.ts`, `action-orchestrator.ts`, `ui-action-tools.ts`, `ui-stability.ts`, `ui-runtime-ios.ts`, `capability-model.ts`, contracts, and tests.

## Capability impact

- Capability category: orchestration / state comparison / primitive evidence
- User-visible or AI-facing behavior change: `stateChanged` in `ActionOutcomeSummary` is now computed via `hasStateChanged()` which explicitly excludes `pageIdentity`. `NavigateBackData.stateChanged` is always `"unknown"` (never `false`); new fields `pageTreeHashUnchanged`, `preBackTreeHash`, `postBackTreeHash` provide evidence for caller judgment. `wait_for_ui_stable` now appears in `describe_capabilities` for both Android and iOS.
- Platforms/frameworks affected: Android / iOS (all platforms using state comparison or navigate_back)
- Support boundary change: none — behavior is hardened, boundaries unchanged
- Invocation guidance impact: none
- Capability truth source checked: `packages/contracts/src/types.ts`, `packages/adapter-maestro/src/capability-model.ts`
- Public docs / canonical guide update: yes — `README.md`, `README.zh-CN.md` tool lists now include `wait_for_ui_stable`

## Implementation notes

- Key implementation changes:
  - Extracted shared `computeTreeHash()` into `ui-tree-hash.ts` with `sampleNodeSignatures()` for InspectUiSummary-derived hashes
  - Created `ui-tree-hash-stability.ts` with `computeStabilityUiTreeHash()` and `flattenNodeSignatures()` for full raw JSON hierarchy snapshots
  - Added `hasStateChanged(a, b)` to `session-state.ts` — a dedicated StateSummary comparator that excludes `pageIdentity`
  - Replaced 4 occurrences of `JSON.stringify` state comparison in `action-orchestrator.ts` with `hasStateChanged()`
  - Weakened `navigate_back` inference: `stateChanged` is always `"unknown"` (never `false`); renamed `noMaterialChange` → `pageTreeHashUnchanged` + `preBackTreeHash` / `postBackTreeHash` evidence fields
  - Fixed `preBackTreeHash` capture timing: now captured BEFORE back action (Android: before `executeUiActionCommand`; iOS: before `tapElementWithMaestroTool`) instead of after
  - Fixed `postBackPageIdentity` to derive from `StateSummary.pageIdentity` (sample-node hash), NOT from `stableFingerprint` (full JSON hash) — both Android and iOS paths
  - Fixed `verifyResolvedIosPointWithHooks` to report mismatch faithfully instead of overriding to `verified: true` when `exitCode === 0`
  - Added `wait_for_ui_stable` to both Android and iOS capability profiles and `ui_inspection` group
- Contract / policy / session / evidence changes: `NavigateBackData.noMaterialChange` replaced with `pageTreeHashUnchanged`, `preBackTreeHash`, `postBackTreeHash`. `StateSummary.pageIdentity` and `PageIdentity` JSDoc updated with explicit non-cross-comparability warnings. `postBackPageIdentity` now comes from StateSummary derivation (same source as `preBackTreeHash`/`postBackTreeHash`).
- Deterministic path and fallback behavior: unchanged — this PR hardens evidence semantics without changing action paths
- AI-first capability guideline consulted: yes — changes are contract-level hardening, not new capability surface
- Compatibility, migration, or rollout notes: `NavigateBackData.noMaterialChange` → `pageTreeHashUnchanged` is a breaking field rename; callers should not have relied on `stateChanged: false` from navigate_back (it was an over-inference). `verifyResolvedIosPointWithHooks` now returns `verified: false` for node mismatches even when tap succeeded — callers that relied on the implicit override need to handle navigation transitions explicitly.

- Rollback strategy: Revert commit — old `JSON.stringify` paths and `noMaterialChange` field remain in git history.
